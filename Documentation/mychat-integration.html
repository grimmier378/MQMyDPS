MQMyChat Integration

Last Updated: 2026-04-20

Overview

MQMyDPS can optionally send battle summaries and session reports to MQMyChat, a separate MacroQuest plugin that provides a custom floating chat window with named channels. When MQMyChat is loaded alongside MQMyDPS, all DPS output is automatically routed to a dedicated "MyDPS" channel in MQMyChat, keeping your regular EQ chat windows clean while still capturing structured combat data.

This integration is completely optional. MQMyDPS checks for MQMyChat at login time and gracefully does nothing if it is absent. There is no build-time dependency and no errors if MQMyChat is not present.

How Detection Works

The MQMyChat availability check happens in LoadCharacterSettings(), which is called each time you enter the game world (SetGameState(GAMESTATE_INGAME)). It calls MacroQuest's IsPluginLoaded("MQMyChat") function and stores the result in a private member variable:

m_myChatLoaded = IsPluginLoaded("MQMyChat");

This cached flag is what every subsequent call to SendToMyChat() checks before attempting to send anything. The value is refreshed each time the character settings are loaded — which means if you load MQMyChat after logging in, you can reload settings (via the Save/Load buttons in the config window) to pick it up without relogging.

You can check the status at any time by reading the result of IsMyChatLoaded(), which simply returns m_myChatLoaded.

How Sending Works

The SendToMyChat() method constructs a MQMyChat send expression and evaluates it using ParseMacroData():

void MyDPSEngine::SendToMyChat(const std::string& message)
{
    if (!m_myChatLoaded)
        return;

    char szBuffer[MAX_STRING] = {};
    sprintf_s(szBuffer, "${MyChat.Send[MyDPS,%s]}", message.c_str());
    ParseMacroData(szBuffer, MAX_STRING);
}

Breaking this down step by step:

  1. If MQMyChat is not loaded, return immediately. No work is done.
  2. Build a string that looks like a MacroQuest TLO expression: ${MyChat.Send[MyDPS,<message text>]}.
  3. Call ParseMacroData(szBuffer, MAX_STRING). This is the standard MQ2 function that evaluates TLO expressions in-place, substituting the result back into the buffer. By calling it on the ${MyChat.Send[...]} expression, we are invoking MQMyChat's Send method with MyDPS as the channel name and the message as the content.
  4. MQMyChat receives the call, auto-creates the "MyDPS" channel if it does not exist, and appends the message to that channel's display.
What is ParseMacroData? ParseMacroData(buffer, size) is a MacroQuest core function that takes a string containing ${} expressions, evaluates each TLO reference, and replaces the expression text with the result. It is normally used internally by the macro engine, but plugins can call it directly to invoke TLO side-effect methods like MyChat.Send. This is the canonical pattern for one plugin to call another plugin's TLO methods at runtime without a compile-time dependency.

MQ Color Codes

All messages sent to MQMyChat use MacroQuest color codes embedded in the message string. These are two-character sequences starting with a backslash that MQ2 chat windows interpret as color changes. MQMyDPS uses the following codes in its messages:

CodeColorUsed for
\atTealPlugin name prefix [MQMyDPS], battle label
\ayYellowDPS values
\agGreenTotal damage amounts
\awWhiteBattle numbers, hit counts, duration, heal counts
\arRedDamage target names (in per-target lines)
\aoOrange"Healed" label in per-heal-target lines
\apPurpleSpawn ID numbers in target labels
\axDefaultReset to default color after a colored segment

The \ax code resets the color to the default chat text color. It is placed after every colored segment to ensure subsequent text returns to normal. Without it, the color would bleed into the next line.

What Gets Sent and When

On /mydps report

When the user runs /mydps report, the command handler builds a message and sends it to both EQ chat and MQMyChat:

std::string report = fmt::format(
    "\at[MQMyDPS]\ax Session DPS: \ay{:.1f}\ax | Total: \ag{}\ax | Hits: \aw{}\ax",
    dps, FormatNumber(g_dpsEngine->sessionDamage), g_dpsEngine->sessionHitCount);
WriteChatf("%s", report.c_str());        // EQ chat
g_dpsEngine->SendToMyChat(report);       // MQMyChat

Example output in MQMyChat:

[MQMyDPS] Session DPS: 1245.3 | Total: 4.2m | Hits: 3381

(With teal, yellow, green, and white coloring on the respective segments.)

On Battle Finalization

When FinalizeBattle() is called (after the battle end delay expires), MQMyDPS sends a multi-line summary to MQMyChat. The first line is the battle overview, followed by one line per damage target, followed by one line per heal target.

Battle overview line:

\at[MQMyDPS]\ax Battle \aw#1\ax ended: DPS \ay1245.3\ax | Dmg \ag4.2m\ax | Hits \aw3381\ax | Dur \aw45s\ax

Per damage-target lines (one per entry in battle.targets):

  \ara skeleton (\ap#1234\ax)\ax - DPS \ay892.1\ax | Dmg \ag40.1k\ax | Hits \aw45\ax

The spawn ID is shown in purple and only appears when the spawn ID is positive (a real resolved ID). Synthetic negative IDs are not shown.

Per heal-target lines (one per entry in battle.healTargets):

  \aoHealed\ax \atGrimmier\ax - \ag12.5k\ax (\aw8\ax heals)

These lines are sent inside the loop over battle.healTargets. They appear in MQMyChat only if at least one heal was recorded for the battle.

The "MyDPS" Channel

The channel name passed to MyChat.Send is always the literal string MyDPS. MQMyChat auto-creates this channel the first time a message is sent to it, so there is no setup required by the user. The channel persists in MQMyChat as long as MQMyChat is loaded.

Because all output goes to a dedicated channel rather than the general EQ chat window, you can configure MQMyChat to display the "MyDPS" channel in a separate floating window positioned wherever is convenient (e.g., near your hotbars or in a corner of the screen) without cluttering your regular chat tabs.

No Output If MQMyChat Is Not Loaded

Every code path that calls SendToMyChat() is guarded by the m_myChatLoaded check inside the method itself. If MQMyChat is absent:

  • Battle finalization still works normally and saves data to history.
  • /mydps report still prints to EQ chat as usual.
  • No error messages are generated.
  • No performance cost is incurred beyond the single boolean check.

Refreshing the MQMyChat Status

If you load MQMyChat after you have already logged in, MQMyDPS will not automatically detect it because the m_myChatLoaded flag is only set during LoadCharacterSettings(). To refresh it without relogging:

  • Open the config window with /mydps config.
  • Click the Load button. This calls LoadCharacterSettings() again, which re-checks IsPluginLoaded("MQMyChat") and updates the cached flag.

Beginner Note: Inter-Plugin Communication in MQ2

MQMyDPS's approach of calling ParseMacroData("${MyChat.Send[...]}") is the standard pattern for one plugin to invoke another plugin's functionality without a compile-time dependency. Here is why this works:

  • MacroQuest's TLO system is a runtime registry. When MQMyChat loads, it calls AddTopLevelObject("MyChat", ...), making ${MyChat} available to the macro engine.
  • ParseMacroData is the same function the macro engine uses when it encounters ${...} in a macro line. Calling it directly from C++ code in another plugin is exactly the same as having a macro script evaluate ${MyChat.Send[MyDPS,hello]}.
  • MQMyDPS does not need to #include any MQMyChat header or link against MQMyChat's DLL. The entire integration happens through the string-based TLO dispatch. If MQMyChat is not loaded, the ${MyChat} TLO simply does not exist and IsPluginLoaded returns false, so the code path is never reached.

This pattern is broadly useful for any plugin that wants to optionally integrate with another without creating a hard dependency. The same approach could be used to call any other plugin's TLO methods at runtime.
