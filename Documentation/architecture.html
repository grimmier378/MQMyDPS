Architecture

Last Updated: 2026-04-20

High-Level Design

MQMyDPS separates responsibilities into four distinct layers: an engine that owns all state, a parser that only understands text patterns, a renderer that only reads state, and a TLO layer that only exposes state. Nothing in the renderer or TLO layer ever writes to the engine — they are read-only consumers. This separation makes the code easier to reason about: if the UI looks wrong, the bug is in the renderer; if a damage number is wrong, the bug is in the engine or parser.

MyDPSEngine is the central class that owns all mutable state. Every game frame, MacroQuest calls OnIncomingChat with combat text. The engine feeds that line to MyDPSParser, which returns a structured DamageRecord if the line matched any known pattern. The engine then resolves which specific mob was hit (by spawn ID), accumulates the damage or heal into per-spawn or per-player entries, and manages battle transitions. Separately, every 250ms, OnPulse monitors the casting state machine and checks whether combat has ended. Finally, every render frame, OnUpdateImGui asks MyDPSRenderer to draw the current state.

OnIncomingChat(line) | v MyDPSEngine::ProcessChat() |-- Update pet name from pLocalPlayer->PetID (pets can be resummoned mid-session) |-- Apply visibility filters (showMyMisses, showHitMe, showDS, etc.) | v MyDPSParser::Parse(line) --> std::optional<DamageRecord> | v MyDPSEngine::RecordDamage() |-- If damage-out (not miss, not incoming, not heal): enter/continue battle |-- ResolveSpawnID() --> pTarget / WhoFollowing / DoT registry / XTarget / synthetic |-- Update sessionDamage, battleDamage counters |-- Update battlePetDamage if PetMelee or PetNonMelee |-- Update battleNonMeleeDmg if NonMelee |-- Update currentTargets[spawnID] --> TargetDamageData | |-- If DirectHeal and inCombat: | battleDirectHeals += damage | currentHealTargets[targetName] --> HealTargetData | |-- If CritHeal and inCombat: | battleCritHeals += damage | |-- If targetSpawnID still 0 and event is incoming (HitBy/HitByNonMelee/MissedMe/DirectHeal/CritHeal): | assign pLocalPlayer->SpawnID (anchor FCT to your character) | |-- fctManager.AddEntry(record, settings) --> floating combat text entry |-- Append to recentRecords deque | v OnPulse (throttled to 250ms) |-- TrackDoTCasting() --> monitor CastingData.SpellID transitions |-- UpdateCombatState() --> GetCombatState() == eCombatState_Combat |-- CleanExpiredRecords() | v OnUpdateImGui |-- RenderMainWindow() --> tabs: Current Battle, History, Targets, Healing, Graphs |-- RenderConfigWindow() --> separate data-driven settings window |-- RenderCombatSpam() --> floating overlay (newest-at-top, click-through option) |-- RenderFloatingText() --> bone-anchored FCT with pop/float/fade animation | v FinalizeBattle() (when battle ends after delay) |-- Creates BattleData with currentTargets (std::move) and currentHealTargets (std::move) |-- Sends colored summary to MQMyChat if loaded |-- Pushes to battleHistory

Why This Data Flow Order Matters

The order of operations in RecordDamage is deliberate. The spawn ID is resolved before the damage is accumulated, because the resolution reads live game state (your current target, your pet's target) that could theoretically change by the next frame. Resolving immediately when the chat line arrives gives the most accurate result.

Heal events are tracked separately from damage. When a DirectHeal record arrives, it is recorded into currentHealTargets keyed by the player name from the chat line — not by spawn ID. Player names are stable and unique, so no spawn resolution is needed. The heal is only counted if inCombat is true; heals cast outside of an active battle do not contribute to battle statistics.

The recentRecords deque is appended to regardless of event type — it stores all events including incoming damage, misses, and heals, because the combat spam overlay shows everything the user configured to see. The battle damage counters, however, only update on outgoing damage events that land (not misses, not incoming).

File Dependencies

MyDPSData.h (structs only, no dependencies beyond mq/Plugin.h for ImVec4) ^ | MyDPSFloatingText.h (includes MyDPSData.h, FCTEntry struct + FCTManager class) ^ | MQMyDPS.h (includes MyDPSData.h + MyDPSFloatingText.h, declares MyDPSEngine) ^ |--- MyDPSParser.h (includes MyDPSData.h) |--- MyDPSRenderer.h (forward declares MyDPSEngine, owns m_lastFCTUpdate) |--- MyDPSTLO.h (includes mq/Plugin.h) | MQMyDPS.cpp (includes everything, plugin entry point, MyDPSEngine implementation)

MyDPSData.h is intentionally kept dependency-free (aside from mq/Plugin.h for ImVec4). It only uses standard library types. This means any file can include it without pulling in all of MacroQuest's headers, which speeds up compilation and prevents accidental coupling.

Key Classes

MyDPSEngine

Central engine class. Owns all mutable state. All public fields are intentionally public because MyDPSRenderer and MyDPSTLO need read access to them without the overhead of getter functions for every field.

  • currentTargets — unordered_map<int, TargetDamageData> keyed by spawn ID. Positive IDs are real spawn IDs from game state, negative IDs are synthetic fallbacks for unresolved spawns. Cleared and re-populated at the start of each new battle.
  • currentHealTargets — unordered_map<string, HealTargetData> keyed by player name. Tracks who you healed during the current battle. Cleared at battle start and moved into BattleData at finalization.
  • battleHistory — vector<BattleData> of completed battles, each owning its own targets and healTargets maps moved from the live maps at battle end.
  • recentRecords — deque<DamageRecord> for the combat spam overlay. Front-popped when records age out past displayTime seconds.
  • activeDoTs — vector<ActiveDoT> tracking which DoT spells are active on which spawn IDs. Entries expire after 5 minutes and are removed when a spawn despawns or the player zones.
  • battleDirectHeals / battleCritHeals — Battle-level counters for direct heal damage and critical heal bonus amounts, accumulated during combat and snapshotted into BattleData at finalization.
  • battlePetDamage / battleNonMeleeDmg — Battle-level counters for pet damage (PetMelee + PetNonMelee combined) and non-melee spell damage (NonMelee type). These feed into BattleData at finalization.
  • fctManager — FCTManager instance that owns the floating combat text ring buffer. Receives entries from RecordDamage, updated and rendered each frame by MyDPSRenderer::RenderFloatingText(). Cleared on zone and on reset.
  • Session and battle accumulators (damage, hits, crits, DoTs).
  • m_myChatLoaded — cached result of IsPluginLoaded("MQMyChat"), set once during LoadCharacterSettings(). Guards all calls to SendToMyChat().

MyDPSParser

Stateless parser except for the character name and pet name it needs to recognize "your" events. Receives a chat line, tries 14 TryParse* methods in frequency order (most common types first), returns std::optional<DamageRecord>. Uses string_view::find() for all pattern matching — no regex (MSVC's regex is slow), no Blech (the MQ2 event system), because combat messages have fixed formats that simple string scanning handles perfectly.

Pet methods are skipped entirely if the pet name is empty (no pet summoned), saving the cost of those checks on most lines. The 11th method, TryParseDirectHeal, handles two EQ heal message formats: the basic "You have healed X for N points." and the spell-name variant "You have healed X for N hit points with your Spell Name."

MyDPSRenderer

Read-only view of engine state. Makes four render calls per frame: main report window (5 tabs), config window, combat spam overlay, and floating combat text. Uses ImPlot for graphs. All three windowed render calls implement a save-on-close pattern that persists settings when the user closes a window via its X button.

The renderer holds its own per-session state:

  • m_showPieChart (default: false) — Toggle for the pie chart
  • m_showLineGraph (default: true) — Toggle for line graphs
  • m_showBarChart (default: true) — Toggle for the bar chart
  • m_graphOffset (default: 0) — Scroll position for the scrollable graph window, which displays 50 battles at a time and can scroll back through up to 250 battles of history
  • m_lastFCTUpdate — Timestamp of the last FCT update, used to compute delta-time for the floating combat text animation system

All three chart types (line, bar, pie) can be shown simultaneously via independent checkboxes. These flags are not persisted to the INI — they reset to defaults each session.

A key pattern used throughout the renderer is the color lookup lambda:

auto C = [&](const char* key) -> ImVec4 {
    auto it = engine.settings.damageColors.find(key);
    return it != engine.settings.damageColors.end() ? it->second : ImVec4(1, 1, 1, 1);
};

This lambda is defined locally inside each rendering function that needs colors. It avoids repeating the map lookup boilerplate on every colored text call. If a key is not found, it falls back to white so nothing crashes.

The config window uses a data-driven layout: arrays of BoolEntry, IntEntry, and FloatEntry structs are iterated to render checkboxes and sliders without repeating the same layout code for every setting. The column count in each config section is computed dynamically from the window width.

Font scaling is applied via ImGui::PushFont(nullptr, FontSizeBase * scale) on both the main window and config window, so the entire UI scales uniformly.

MyDPSType / MyDPSTargetType

MQ2Type subclasses that expose engine data as TLO members. MyDPSType is the top-level ${MyDPS} object. MyDPSTargetType represents individual target entries accessed via ${MyDPS.Target[id]}. The target type stores only the spawn ID in VarPtr.Int and looks up the live TargetDamageData from currentTargets on every member access, so it always reflects current data.

Accumulated healing for one player across the current battle. Keyed by player name in the currentHealTargets map (and in BattleData::healTargets after finalization). Heals are per player, not per NPC target — there is no spawn ID for heal targets because you are healing players, not mobs.

ActiveDoT

Tracks a DoT spell cast so that later tick messages can be attributed back to the correct spawn. Fields: targetSpawnID (resolved at cast time), targetName (for matching incoming tick text), spellName (for matching specific spells), castTime (for 5-minute expiry).

MyDPSSettings

All user-configurable options. Notable fields:

  • spamClickThrough (default true): causes the spam overlay to use ImGuiWindowFlags_NoInputs, passing clicks through to the game below.
  • damageColors: an unordered_map<string, ImVec4> mapping color keys to RGBA values. Includes keys for damage types, stat display colors, and entity name colors. See the full color key table below.
  • showFCT (default false): master toggle for floating combat text.
  • Per-type FCT toggles: showFCT_Melee, showFCT_DD, showFCT_DoT, showFCT_Pet, showFCT_Crit, showFCT_Heals, showFCT_CritHeals, showFCT_HitBy (all default true).
  • fctBaseFontSize (default 24.0): base pixel size for FCT text.
  • fctFontScale (default 1.5): multiplied with base font size for final size.
  • fctShadowOffset (default 2.0): pixel offset for the drop shadow, scaled proportionally to font size.
  • fctLifetime (default 2.5s): how long each FCT entry floats before fading out.

Color key resolution differs between combat spam and FCT. Floating Combat Text (FCT) System

The FCT system renders damage and heal numbers anchored to the 3D position of the target spawn, projected onto the 2D screen. Numbers pop in enlarged (1.4x), float upward (150px), and fade out after 60% of their lifetime. The system uses a 64-entry ring buffer managed by FCTManager, with bone-anchored 3D-to-screen projection via eqlib/graphics headers.

Key aspects: a 5-stage filter chain in AddEntry (master toggle, spawn ID, miss, damage, per-type toggle), pet color override to "pet" key, alternating fan spread for rapid hits, head bone projection with avatar-height fallback, and drop shadow rendering to ImGui's background draw list.

For comprehensive details on the FCT architecture, animation formulas, projection math, entry creation, type filtering, and configuration, see the dedicated Floating Combat Text page.

Save-on-Close Window Behavior

All three closeable windows (Main, Config, Combat Spam) implement a save-on-close pattern. Before drawing, each caches its open state. After ImGui::End(), if the window was open before but is now closed (user clicked the X button), it calls engine.SaveCharacterSettings(). This ensures the window visibility state is persisted to the INI immediately when the user closes a window.

bool wasOpen = engine.showMainWindow;
if (ImGui::Begin(windowName.c_str(), &engine.showMainWindow, ...))
{ /* ... */ }
ImGui::End();
if (wasOpen && !engine.showMainWindow)
    engine.SaveCharacterSettings();

This pattern is repeated identically for showConfigWindow and showCombatSpam.

OnBeginZone: Save Before Clear

When the player begins zoning, OnBeginZone saves all current settings to disk before clearing transient state. This ensures that any setting changes made during the session (including window positions saved by ImGui) are not lost. After saving, it sets inCombat = false, clears activeDoTs, and clears the FCT manager. Battle history and session data are preserved across zones.

Complete Data Flow: Healing Example

To make the heal tracking flow concrete, here is what happens when you cast Celestial Remedy on your tank and it heals for 2500:

  1. EQ sends chat: "You have healed Grimmier for 2500 hit points with your Celestial Remedy."
  2. OnIncomingChat fires. ProcessChat calls m_parser->Parse().
  3. TryParseDirectHeal matches because the line starts with "You have healed ". It extracts targetName="Grimmier", damage=2500, and spellName="Celestial Remedy". Returns a DamageRecord with type=DirectHeal, attackVerb="heal".
  4. In RecordDamage: the record is not flagged as isDamageOut (heals are excluded from that check), so it does not start or continue a battle by itself. If inCombat is true:
    • battleDirectHeals += 2500
    • currentHealTargets["Grimmier"] is found or created, directHeals += 2500, healCount++
  5. The record is appended to recentRecords and shown in the combat spam overlay with the "heal" color key (green).
  6. At battle end, FinalizeBattle() moves currentHealTargets into battle.healTargets. The Healing tab aggregates this across battles.
