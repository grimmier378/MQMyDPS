ImGui Rendering

Last Updated: 2026-04-22

What Is Immediate-Mode GUI?

ImGui (Dear ImGui) is an "immediate mode" GUI library, meaning there are no persistent UI objects or state stored in the GUI layer. Instead, you call functions like ImGui::Text() and ImGui::Button() every render frame, and ImGui draws them and reports input. If a window is invisible, you simply do not call its render function that frame — there is nothing to hide or show, you just stop or start drawing it.

This model is well-suited for a DPS tracker because the data changes constantly and you want the display to always reflect the current state without maintaining a parallel UI data structure. Every frame, MyDPSRenderer reads whatever is currently in MyDPSEngine and draws it. No synchronization needed.

Four Render Calls

MyDPSRenderer makes four render calls per frame. Three are windowed ImGui windows that check their own visibility flag; the fourth is the floating combat text system that draws directly to the background draw list:

Render CallToggleVisibility flagDescription
RenderMainWindow /mydps or show/hide engine.showMainWindow Main window with menu bar and 5 tabs. Font scaled by fontScale.
RenderConfigWindow /mydps config or Config menu item engine.showConfigWindow Separate settings window with data-driven layout. Font scaled by fontScale.
RenderCombatSpam /mydps spam or "Show Combat Output" checkbox engine.showCombatSpam && engine.tracking Floating overlay with recent events newest-at-top. Requires both visibility flag and tracking to be active. Font scaled by spamFontScale.
RenderFloatingText /mydps fct or "Enable FCT" checkbox engine.settings.showFCT Bone-anchored floating combat text with pop/float/fade animation. Draws to background draw list.

Theme Wrapping

All three windowed render calls (Combat Output, MainWindow, ConfigWindow) apply the user-selected theme before ImGui::Begin and reset it after ImGui::End:

ImGuiStyle oldStyle = ImGuiTheme::ApplyTheme(settings.themeIdx);
// ... ImGui::Begin / End ...
ImGuiTheme::ResetTheme(oldStyle);

RenderFloatingText does not use theme wrapping because it draws directly to the background draw list rather than creating an ImGui window.

Save-on-Close Pattern

All three closeable windows (Main, Config, Combat Output) implement a save-on-close pattern. Before drawing, the renderer caches the window's open state. After ImGui::End(), if the window transitioned from open to closed (user clicked the X button), it calls engine.SaveCharacterSettings() to persist the updated visibility state to the INI file immediately.

bool wasOpen = engine.showMainWindow;
if (ImGui::Begin(windowName.c_str(), &engine.showMainWindow, ...))
{ /* ... */ }
ImGui::End();
if (wasOpen && !engine.showMainWindow)
    engine.SaveCharacterSettings();

This pattern is repeated identically for all three windows. The RenderFloatingText call does not have this pattern because it is not a windowed element.

The Color Lookup Lambda

Every rendering function that needs to color text defines a local lambda named C (or Clr in the History tab). This is a key code pattern used throughout the renderer:

auto C = [&](const char* key) -> ImVec4 {
    auto it = engine.settings.damageColors.find(key);
    return it != engine.settings.damageColors.end() ? it->second : ImVec4(1, 1, 1, 1);
};

With this lambda, coloring a text call looks like:

ImGui::TextColored(C("dps"), "%.1f", battleDPS);

The lambda does a hash map lookup in settings.damageColors and returns white as a safe default if the key is missing. This pattern prevents verbose repeated lookups and centralizes the fallback behavior. The lambda captures the engine reference by reference ([&]), so it always reads the current settings.

Font Scaling

Both the main window and config window push a scaled font before drawing any content, and pop it after. This scales all text rendered between the push and pop calls:

ImGui::PushFont(nullptr, ImGui::GetStyle().FontSizeBase * engine.settings.fontScale);
// ... all window contents ...
ImGui::PopFont();

Passing nullptr as the first argument means "use the default font, just at a different size". FontSizeBase is the base size of the current font. Multiplying it by fontScale (range 0.5–2.0) scales the entire window's text proportionally. The Combat Output overlay uses spamFontScale the same way, allowing the overlay to be sized independently from the main window.

Main Window

Window Identity

The window title includes the character name:

auto windowName = fmt::format("DPS Report##{}", engine.charName);

ImGui uses window titles as unique IDs. The ## separator means everything after it is part of the ID but not shown in the title bar. Including the character name means two MQMyDPS instances (e.g., running in two EQ windows) have separate ImGui windows that do not conflict.

Menu Bar

The main window has a menu bar created with ImGui::BeginMenuBar(). Unlike a full dropdown menu, the items here are rendered inline as a compact single-row bar. Each item uses ImGui::MenuItem():

  • Start/Stop — Toggles engine.tracking. Shows "Stop" when tracking is active, "Start" when it is not. Uses Material Design icon font characters (ICON_MD_STOP, ICON_MD_PLAY_ARROW) before the label.
  • Reset — Calls engine.ResetAll().
  • Config — Toggles the config window.
  • DPS: X.X — Live session DPS, colored using the "dps" color key (gold by default). Updated every frame.
  • [COMBAT] — Shown in red when engine.inCombat is true.
DPS counter color: The menu bar DPS display uses ImGui::TextColored(dpsColor, "DPS: %.1f", ...) where dpsColor is looked up from engine.settings.damageColors["dps"], defaulting to gold (1.0, 0.9, 0.4) if not found.

Tab Bar (5 Tabs)

The main window contains a tab bar with five tabs. This is the current tab order:

  1. Current Battle
  2. History
  3. Targets
  4. Healing
  5. Graphs

All five tab contents are wrapped in BeginChild/EndChild with ImVec2(0, 0) size, so the tab bar stays fixed at the top and each tab's content scrolls independently.

Tab: Current Battle

If no battle is active and no battle damage has been recorded, a hint text is shown. Otherwise, the tab renders an adaptive-column layout using a StatEntry struct array. Each stat is a compact "Label: Value" pair with the value colored by its data type. The number of columns adapts to the window width (std::max(1, sizeX / 150)), flowing stats into multiple columns on wide windows and wrapping to fewer on narrow ones. The stats are:

Table flags: The History table uses ImGuiTableFlags_Hideable | ImGuiTableFlags_Reorderable | ImGuiTableFlags_ScrollY in addition to the standard RowBg | Borders | Resizable flags. This means you can right-click the header row to show/hide any column except Battle# (which has the NoHide flag), and you can drag column headers to rearrange them. The ScrollY flag enables vertical scrolling when many battles are recorded.

Each row represents one completed battle. The Battle# column contains a TreeNode whose label reads "Battle (N) NPC (N) Healed (N)", showing the battle number, the number of damage targets, and the number of healed players at a glance. The tree node label format from the source:

ImGui::TreeNode("##battle", "Battle (%d) NPC (%d) Healed (%d)",
    b.battleNumber, static_cast<int>(b.targets.size()), static_cast<int>(b.healTargets.size()));

Sort order is controlled by settings.sortNewest. When newest-first, iteration starts from the last element of battleHistory and steps backward. When oldest-first, it goes forward. Explicit integer indexing is used for backward iteration because a range-for loop cannot easily iterate in reverse.

Tab: Targets

Two sections separated by a horizontal line:

  • Current Battle Targets: A 4-column table (Name, DPS, Hits, Total) reading live from engine.currentTargets. Updated every frame. Shown only if the current battle has any targets.
  • All Session Targets: A 4-column table (Name, Hits, Avg, Total) reading from the pre-sorted sortedSessionTargets vector. This vector is maintained incrementally and re-sorted on a 250ms throttle, rather than aggregating inside the render function every frame. Results are sorted by total damage descending. The table uses ImGuiTableFlags_ScrollY for vertical scrolling when many targets have been recorded across the session.

Tab: Healing

The Healing tab reads from the pre-sorted sortedSessionHeals vector, which aggregates heal data from all completed battles and the current in-progress battle into a unified per-player collection. This vector is maintained incrementally and re-sorted on a 250ms throttle, rather than re-aggregating inside the render function every frame. This means it shows your total healing output for the entire session, not just one battle.

The results are sorted by GetTotalHeals() descending so the player you healed the most appears at the top.

The table has 5 columns, all hideable and reorderable except "Player" which is marked NoHide:

ColumnDescription
PlayerPlayer name (NoHide). Name healed.
Direct HealsSum of all DirectHeal amounts for this player
Crit HealsSum of all CritHeal bonus amounts associated with this player
CountNumber of direct heal cast events
TotaldirectHeals + critHeals combined

Below the table, the Healing tab also shows a pie chart labeled "Healing Distribution by Player". This uses the same ImPlot PlotPieChart approach as the damage pie chart in the Graphs tab, with each player's total heals as their slice value.

Heal counts vs crit heals per player: healCount in HealTargetData counts only DirectHeal events, not CritHeal events. Critical heal messages do not include a target name in EQ's chat format, so they cannot be attributed to a specific player. The critHeals field in HealTargetData is present in the struct but is currently zero for all players because crit heals are only accumulated at the battle level (battleCritHeals), not per player.

Tab: Graphs

Three independent checkboxes at the top of the tab let you toggle each chart type on or off. All three can be shown simultaneously, or any combination:

  • DPS Graph (m_showDpsGraph, default: on)
  • Damage Breakdown (m_showDmgGraph, default: on)
  • Bar Chart (m_showBarChart, default: on)
  • Pie Chart (m_showPieChart, default: off)

These toggle flags are members of MyDPSRenderer (not the engine), so they are per-session and not persisted to the INI file.

The Graphs tab content is wrapped in a child window using ImGuiChildFlags_Borders and PushStyleVar(ImGuiStyleVar_WindowPadding, ImVec2(8.0f, ...)) to provide scroll gutters on both sides of the content area.

Graph data vectors are cached as renderer members and only rebuilt when the battle count or scroll offset changes, avoiding per-frame vector reconstruction.

Scrollable Graph Window

When there are more battles than the display window can show at once, a slider appears to let you scroll back through history. Two constants control this:

static constexpr int GRAPH_WINDOW   = 50;   // battles visible at once
static constexpr int GRAPH_MAX_BACK = 250;  // max history depth

Each graph (DPS, Damage Breakdown, Bar Chart) has its own independent scroll bar and its own GraphScrollState struct (m_dpsScroll, m_dmgScroll, m_barScroll). Each scroll state tracks an offset and an autoScroll flag. When autoscroll is on (default), the graph snaps to the most recent battles. If the user drags the scroll bar away from the end, autoscroll disengages. Scrolling back to the end re-engages autoscroll. Graph data vectors are cached as renderer members and rebuilt only when the total battle count changes (not when offset changes), covering the full GRAPH_MAX_BACK range.

Line Graph: DPS by Battle

A single-series line chart plotting DPS (Y axis) against battle number (X axis). The line color uses the "dps" color key (gold by default). Y axis is constrained to a minimum of 0 to prevent negative space.

Line Graph: Damage Breakdown by Battle

A multi-series line chart with three independent Y axes, each auto-scaling to fit its own data range:

Y AxisPositionLabelSeriesColor Keys
Y1Left"Damage" Melee, DD, DoT, Pet "hit", "non-melee", "dot", "pet"
Y2Right"Crits" Crit "crit"
Y3Right"Heals" Heals, Crit Heals "heal", "critHeals"

All three Y axes are constrained to a minimum of 0. The three-axis design means damage values, crit values, and heal values each auto-scale independently, so a small heal amount is still visible even when damage values are much larger. The legend is positioned in the top-right corner and is clickable — clicking a series name in the legend hides or shows that series.

The "Melee" series is computed as total damage minus crits, DoTs, DD, and pets (with a floor of 0), so it represents the remaining melee-only portion of damage.

A grouped bar chart with 7 thin bars side-by-side per battle. Each bar represents a different damage or heal category:

Bars are grouped using X-offset positioning. The total group width is 0.8 units, divided equally among 7 bars. Each bar is shifted left or right from the battle number's X position so they sit side-by-side without overlapping. Bar fill colors come from damageColors using the same color keys as the line graph. The legend is clickable to hide or show individual series.

Pie Chart: Damage Distribution by Target

Aggregates total damage by target name across all completed battles and the current battle. Entries are sorted by damage descending before being passed to ImPlot, so the largest slice always starts at the top. Uses ImPlotFlags_Equal to ensure the pie renders as a circle rather than an ellipse.

All plots use axis constraint calls to lock the minimum at 0:

ImPlot::SetupAxisLimitsConstraints(ImAxis_Y1, 0, HUGE_VAL);

This prevents the axes from ever going negative, which would be visually confusing for a damage chart.

Combat Output Overlay

Window Flags

ImGuiWindowFlags flags = ImGuiWindowFlags_NoTitleBar | ImGuiWindowFlags_NoFocusOnAppearing;
if (engine.settings.spamClickThrough)
    flags |= ImGuiWindowFlags_NoInputs;

The Combat Output window is titled "Combat Output##CharName" and is always borderless and never steals focus when it first appears. When spamClickThrough is true (the default), ImGuiWindowFlags_NoInputs is added. This flag tells ImGui to pass all mouse input (clicks, hover, scroll) through the window to whatever is underneath it. The result is that you can click on EverQuest's game world through the Combat Output overlay without it intercepting any mouse events.

When spamClickThrough is false, the overlay is a normal interactive window that you can drag by clicking its body. This is useful if you want to reposition it, but during active combat it means your left clicks on monsters may accidentally drag the overlay instead.

The /mydps lock command toggles spamClickThrough at runtime. The "Lock Combat Output" checkbox in the config window does the same thing.

Background Transparency

ImGui::SetNextWindowBgAlpha(engine.settings.bgColor.w);

The background alpha is taken from the W component (alpha) of bgColor. The default value is 0.5, giving a semi-transparent background. Setting it to 0.0 makes the overlay fully transparent (just floating text). Setting it to 1.0 gives a fully opaque background.

Newest-at-Top Rendering

The Combat Output overlay iterates engine.recentRecords using a reverse iterator (rbegin to rend), so the most recent event always appears at the top of the overlay. Events older than displayTime seconds are skipped:

for (auto it = engine.recentRecords.rbegin(); it != engine.recentRecords.rend(); ++it)
{
    const auto& rec = *it;
    auto age = std::chrono::duration_cast<std::chrono::seconds>(now - rec.timestamp).count();
    if (age > engine.settings.displayTime)
        continue;
    // render the record
}

Records are not removed from the deque during rendering — cleanup is handled separately by CleanExpiredRecords() in OnPulse. The render loop simply skips records that are too old to display.

Content Rendering

The color for each line is looked up from settings.damageColors using the attack verb for melee and pet melee events, and the DamageTypeToColorKey() result for all others. If no matching key exists, white is used. DirectHeal events use the "heal" key (green).

The text format depends on which display options are enabled:

showTypeshowTargetExample output
truetrue[crush] a skeleton: 150
truefalse[crush] 150
falsetruea skeleton: 150
falsefalse150

Miss events show "MISS" instead of a damage number. Heal events show the healed player's name and the heal amount when both show options are active.

Config Window

The config window has a default size of 350x350 pixels.

Data-Driven Layout

Instead of writing separate ImGui calls for every setting, the config window defines a BoolEntry struct array at the top of RenderConfig():

struct BoolEntry  { const char* label; bool* value; };

BoolEntry bools[] = {
    { "Show Type",         &s.showType },
    { "Show Target",       &s.showTarget },
    { "Show My Misses",    &s.showMyMisses },
    { "Show Missed Me",    &s.showMissMe },
    { "Show Hit Me",       &s.showHitMe },
    { "Show Crit Heals",   &s.showCritHeals },
    { "Show Damage Shield",&s.showDS },
    { "Track Pet",         &s.addPet },
    { "Show Combat Output", &engine.showCombatSpam },
    { "Sort Newest First",  &s.sortNewest },
    { "Auto Start",         &s.autoStart },
    { "Lock Combat Output", &s.spamClickThrough },
};

Note that "Show Combat Output" points to engine.showCombatSpam directly (not a field on MyDPSSettings), because it is a window visibility flag rather than an option setting. Adding a new boolean setting requires exactly one line in the bools array — the rendering loop handles layout automatically.

Sliders (Display Time, Battle End Delay, Font Scale, Output Font Scale, and the FCT sliders) are direct ImGui::SliderInt / ImGui::SliderFloat calls rather than data-driven structs.

The column count for table-based sections is computed dynamically:

int sizeX = static_cast<int>(ImGui::GetWindowWidth());
int col = std::max(1, sizeX / 165);  // For Display tab

This means when you make the config window wider, more checkboxes appear per row. When you narrow it, they wrap to more rows. std::max(1, ...) ensures there is always at least one column even if the window is very narrow.

Save and Load Buttons

  • Save — Calls engine.SaveCharacterSettings() and then sets engine.showConfigWindow = false, closing the config window.
  • Load — Calls engine.LoadCharacterSettings() to reload settings from disk, discarding any unsaved changes. This also re-checks whether MQMyChat is loaded.

Colors Tab

The Colors tab first renders a ColorEdit4 for the background color (Combat Output overlay background with alpha). Then it shows only the UI-only subset of settings.damageColors — specifically: background, dmg-target, heal-target, dps, duration, avg, and total. Each is rendered as one ColorEdit4 in a table with sizeX / 150 columns, using ImGuiColorEditFlags_NoInputs to show only the colored swatch, keeping the tab compact. All FCT-related colors (melee verbs, crit, dot, non-melee, heal, critHeals, pet, hit, etc.) have moved to the FCT tab's unified type table where they are shown alongside their icon configuration.

FCT Tab

The FCT tab contains:

  • Enable FCT — Master checkbox toggling settings.showFCT.
  • Type filter toggles — A checkbox grid (columns computed dynamically at sizeX / 130) with 12 individual checkboxes: Melee, DD, DoT, Pet, Crit, Heals, Crit Heals, Hit By, DS, Icons, Distinct Melee, and Spell Icons. Each toggle controls visibility of that FCT type. "DS" controls damage shield proc visibility. "Icons" controls whether icons are rendered alongside FCT text. "Distinct Melee" controls whether individual melee verbs appear as separate rows in the type table below. "Spell Icons" enables actual spell icon lookup for spell-based damage types.

Below the master toggle, a unified type table is rendered with 4 columns:

ColumnDescription
TypeThe FCT type name (e.g. Melee, DD, DoT, Pet, Crit, Heals, Crit Heals, Hit By, and all melee verb colors formerly in the Colors tab)

This unified table consolidates all FCT-related colors (previously shown in the Colors tab) with icon configuration in a single location. Each row represents one configurable FCT type, including melee verbs (crush, slash, hit, etc.), crit, dot, non-melee, pet, heal, critHeals, and hit-by types.

Icon Picker Window

Clicking an icon cell in the unified type table opens a separate ImGui window titled "Icon Picker - TypeName##FCTIconPicker". This window contains:

  • Spell/Item tabs — Two tabs selecting between the spell icon atlas and the item icon atlas as the icon source.
  • Scale slider — Controls the display size of icons in the picker grid for easier browsing.
  • Page navigation — A SliderInt labeled "Page" to page through the available icons in the selected atlas. Clicking an icon in the grid selects it for that FCT type and closes the picker.

Icon rendering in both the type table and the picker requires CTextureAnimation objects loaded from the SIDL manager. EnsurePickerAnimations() lazy-loads these on first use, finding A_SpellGems (spell icon atlas) and A_DragItem (item icon atlas) via pSidlMgr->FindAnimation() and cloning them into renderer-owned copies.

Item icons are stored with a +500 offset to distinguish them from spell icons: IDs 0–499 represent spell icons, while IDs 500+ represent item icons (with the actual atlas cell being iconID - 500). This convention is used consistently across the icon picker, the FCT type table rendering, and the override storage in fctIconOverrides.

Floating Combat Text Rendering

RenderFloatingText is the fourth render call made from OnUpdateImGui. Unlike the other three, it does not create an ImGui window. Instead, it computes a delta-time from m_lastFCTUpdate (stored on the renderer), clamps it to 16ms if out of range, then delegates to the FCT manager for update and render passes. The FCT manager draws directly to ImGui::GetBackgroundDrawList(), which renders behind all ImGui windows but on top of the EQ game world.

For full details on the FCT system — including the entry creation filter chain, bone-anchored projection math, animation formulas (pop-in, float, fade), drop shadow rendering, type filtering, and configuration — see the dedicated Floating Combat Text page.
