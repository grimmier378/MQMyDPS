Spawn ID Tracking

Last Updated: 2026-04-23

The Core Problem

EverQuest's combat chat text never includes spawn IDs. A spawn ID is an integer that uniquely identifies one specific mob in the world at a given time. The chat system only knows mob names. When you fight four mobs all named "a skeleton", every melee hit, every DoT tick, and every crit message just says "a skeleton" — there is nothing in the text to tell them apart.

This is a challenge for any DPS tracker. A Lua-based tracker can read spawn IDs through TLOs (e.g. mq.TLO.Target.ID()), but Lua event callbacks only fire when mq.doevents() is called. If the player's target changes between the chat line arriving and the next mq.doevents() cycle, the callback will read the wrong target — attributing damage to the wrong mob. The more frequently mq.doevents() runs, the smaller the timing window, but it can never be fully eliminated in Lua's event-driven model.

MQMyDPS avoids this as a C++ plugin: the OnIncomingChat callback fires synchronously when each chat line arrives, before any target changes can occur. The plugin reads game state (current target, pet target, XTarget list, last cast spell) at the exact moment the chat line is processed, guaranteeing the spawn ID matches the event.

Heal events are different: heals target player characters, not NPCs, and player names are unique within a zone. No spawn ID resolution is needed for heal tracking — the player name from the chat line is used directly as the map key in currentHealTargets.

What Is a Spawn ID?

Every entity in EverQuest (player, NPC, pet, object) has a SpawnID: a 16-bit integer assigned when the entity enters the world and reused when it despawns. In C++, you access this through pTarget->SpawnID on a PlayerClient* pointer. The value is small (typical values are in the range 1–4000 depending on zone), and it is the canonical key for identifying any specific entity in the zone.

Positive spawn IDs are real game IDs. Negative spawn IDs in MQMyDPS are synthetic fallbacks created by the engine when no real ID can be determined — they are assigned from a counter that starts at -1 and decrements.

The Resolution Chain

ResolveSpawnID(record) is called once per damage event, immediately after parsing and before accumulation. It tries strategies in order, returning the first positive ID it finds. If all strategies fail, it returns a negative synthetic ID so the event is still tracked, just grouped under an anonymous entry.

ResolveSpawnID(record) | +-- If targetName is empty: return synthetic ID immediately | (this happens for raw Crit messages before RecordDamage fills the name) | +-- Switch on record.type: | | Melee / NonMelee / Crit: | if pTarget != null AND pTarget->DisplayedName == targetName: | return pTarget->SpawnID [Strategy 1: Direct Target] | | PetMelee / PetNonMelee: | pPet = GetSpawnByID(pLocalPlayer->PetID) | if pPet != null: | pPetTarget = pPet->WhoFollowing | if pPetTarget != null AND pPetTarget->DisplayedName == targetName: | return pPetTarget->SpawnID [Strategy 2: Pet Target] | | DoT: | dotID = ResolveDoTSpawnID(targetName, spellName) | if dotID > 0: return dotID [Strategy 3a: DoT Registry] | if pTarget != null AND pTarget->DisplayedName == targetName: | return pTarget->SpawnID [Strategy 3b: Current Target Fallback] | | DamageShield / default: | (intentionally no type-specific check - fall through to existing entry scan) | +-- Scan currentTargets for existing entry matching targetName: | if found: return existing ID [Strategy 4: Existing Entry] | (prevents creating duplicate entries for the same mob) | +-- ResolveFromXTarget(targetName): | scan pLocalPC->pExtendedTargetList | filter: NPC only, not player pets | if exactly 1 NPC matches: return its SpawnID [Strategy 5: XTarget] | if 0 or 2+ matches: continue | +-- return --m_syntheticIDCounter [Strategy 6: Synthetic Fallback]
Crit path target initialization: When a Crit event is the first hit on a target, the code now properly initializes the TargetDamageData entry with name, spawnID, and firstHit, matching the initialization performed for all other damage types. Previously, if the very first hit on a target was a crit, the target entry would be created with an empty name and an epoch firstHit timestamp, because the crit code path did not include the same new-target initialization logic as the non-crit path.

Strategy 1: Direct Target (Melee, NonMelee, Crit)

if (pTarget && pTarget->DisplayedName == record.targetName)
    return pTarget->SpawnID;

pTarget is a global pointer maintained by MacroQuest that always points to your currently targeted entity, or null if you have no target. DisplayedName is the name shown in the target window (not the internal name, which may differ for some NPCs). This is the most reliable strategy because:

  • EverQuest delivers the chat line in the same frame as the hit lands.
  • A melee hit can only come from your current target — you cannot melee something you are not targeting.
  • The name comparison confirms that the target you have selected is actually the mob named in the chat line (guards against edge cases like a pet hit coinciding with your target switch).

When it fails: If you switch targets the instant a message arrives (extremely rare), pTarget might already point to your new target. The name check prevents misattribution in this case — if names do not match, the strategy falls through rather than assigning the wrong ID.

Strategy 2: Pet Target (PetMelee, PetNonMelee)

PlayerClient* pPet = GetSpawnByID(pLocalPlayer->PetID);
if (pPet != nullptr)
{
    PlayerClient* pPetTarget = pPet->WhoFollowing;
    if (pPetTarget && pPetTarget->DisplayedName == record.targetName)
        return pPetTarget->SpawnID;
}

pLocalPlayer->PetID is the spawn ID of your pet (0 if you have no pet). GetSpawnByID() looks up the PlayerClient* for that ID in the zone's spawn list. WhoFollowing is the pointer to whichever entity the pet is currently attacking or following.

Why not use pTarget for pet hits? Your pet operates independently. It attacks its own target, which is often the same as your target but is not guaranteed to be. In a group fight, your pet might be attacking mob A while you are targeting mob B. Using pTarget would misattribute the pet hit to the wrong mob. The pet's WhoFollowing pointer gives the correct target directly.

When it fails: WhoFollowing is null when the pet has no target (it just killed something, or you commanded it to stay). In that case, the strategy falls through. This can happen briefly right after a mob dies and the pet is in the transition to a new target.

Strategy 3: DoT Cast Registry

DoT (damage-over-time) spells are the most complex case. A DoT tick arrives as:

a skeleton has taken 300 damage from your Envenomed Bolt.

There is no spawn ID in this text. Your current target might be a completely different mob (you often cast a DoT and then switch to another target while the first one ticks). pTarget would give the wrong spawn ID. The DoT registry solves this by recording the target at the moment of casting, before you switch away.

How the Registry Is Built (in TrackDoTCasting)

TrackDoTCasting() runs every 250ms in OnPulse. It watches the player's casting state by monitoring pLocalPlayer->CastingData.SpellID:

  • While CastingData.SpellID != -1: a spell is being cast. Save the spell ID and target ID into m_lastCastSpellID and m_lastCastTargetID on every pulse.
  • When CastingData.SpellID transitions to -1: the cast just finished (or was interrupted). Check if the last cast spell is a DoT (IsDoTSpell()) and detrimental (IsDetrimentalSpell()). If both conditions are true, create an ActiveDoT entry with the target's spawn ID and push it to activeDoTs.
if (pLocalPlayer->CastingData.SpellID != -1)
{
    m_lastCastSpellID  = pLocalPlayer->CastingData.SpellID;
    m_lastCastTargetID = pLocalPlayer->CastingData.TargetID;
}
else if (m_lastCastSpellID != -1)
{
    EQ_Spell* pSpell = GetSpellByID(m_lastCastSpellID);
    if (pSpell && pSpell->IsDoTSpell() && pSpell->IsDetrimentalSpell())
    {
        ActiveDoT dot;
        dot.targetSpawnID = m_lastCastTargetID;
        dot.targetName    = pSpawnTarget->DisplayedName;
        dot.spellName     = pSpell->Name;
        dot.castTime      = std::chrono::steady_clock::now();
        activeDoTs.push_back(std::move(dot));
    }
    m_lastCastSpellID = -1;
}

Why the double condition (IsDoTSpell AND IsDetrimentalSpell)? IsDoTSpell() checks the spell's SPA effects (spell categories 0, 20, 69, 114, 125 cover DoT effects). But some beneficial DoT-like spells exist (HoT — heals over time). The IsDetrimentalSpell() check ensures the registry only captures offensive DoTs, not heals.

How Attribution Works

When a DoT tick arrives and is parsed, ResolveDoTSpawnID(targetName, spellName) searches activeDoTs for an entry where both the target name and spell name match. If found, the registered spawn ID is returned.

for (const auto& dot : activeDoTs)
    if (dot.targetName == targetName &&
        (spellName.empty() || dot.spellName == spellName))
        return dot.targetSpawnID;

Scenario: You cast Envenomed Bolt on skeleton #1234, then switch target to skeleton #5678. Skeleton #1234 takes a tick. The chat says "a skeleton has taken 300 damage from your Envenomed Bolt." The registry has an entry for "a skeleton" + "Envenomed Bolt" = spawnID 1234. Returned correctly, even though your current target is #5678.

Scenario: You have the same DoT on two skeletons. The registry has two entries both with name "a skeleton" + "Envenomed Bolt". The first match wins, which may be wrong. This is an inherent limitation when fighting multiple copies of the same named mob with the same spell. Exact deduplication would require the game to include spawn IDs in tick messages, which it does not.

Registry Cleanup

DoT entries are removed by three mechanisms to prevent the registry from growing indefinitely and causing stale mis-attributions:

  • Mob despawns: OnRemoveSpawn fires when any mob is removed from the world. The handler calls RemoveActiveDoTs(spawnID), erasing all entries for that spawn. This is the most precise cleanup — it happens the moment the mob is gone.
  • Player zones: OnBeginZone saves settings first, then clears the entire activeDoTs vector and FCT entries. All spawn IDs from the old zone are meaningless in the new zone.
  • Time-based expiry: Every 250ms in TrackDoTCasting(), entries older than 5 minutes are removed. This covers edge cases where a mob leaves the zone without triggering OnRemoveSpawn, or where you left combat and the DoT simply stopped ticking without any cleanup event.

Strategy 4: Existing Entry Lookup

for (const auto& [id, data] : currentTargets)
    if (data.name == record.targetName)
        return id;

This strategy runs after all type-specific checks and before XTarget. Its purpose is to avoid creating duplicate entries for the same mob. If you have already seen hits on "a skeleton" with spawn ID 1234 in this battle, and a new hit arrives for "a skeleton" that could not be resolved by direct target or DoT registry, returning 1234 keeps all damage on the same entry rather than creating a new synthetic entry.

This is particularly important for damage shields: the mob that hit you (triggering the DS) may or may not be your current target, but it is almost certainly in your current battle. Using the existing entry prevents fragmentation of the battle summary when DS damage cannot be directly attributed.

Limitation: If you genuinely are fighting two mobs with the same name and only one is in currentTargets so far, this strategy will attribute new hits to the existing entry even if they came from the other mob. The XTarget strategy below can sometimes distinguish them.

Strategy 5: XTarget Scan

for (const ExtendedTargetSlot& slot : *pLocalPC->pExtendedTargetList)
{
    if (slot.XTargetSlotStatus == eXTSlotEmpty || slot.SpawnID == 0)
        continue;
    if (slot.Name != targetName)
        continue;

    SPAWNINFO* pSpawn = GetSpawnByID(slot.SpawnID);
    if (!pSpawn || pSpawn->Type != SPAWN_NPC)
        continue;

    if (pSpawn->MasterID != 0)
    {
        SPAWNINFO* pMaster = GetSpawnByID(pSpawn->MasterID);
        if (pMaster && pMaster->Type == SPAWN_PLAYER)
            continue;  // Skip player-owned pets
    }

    matchID = pSpawn->SpawnID;
    matchCount++;
}
if (matchCount == 1)
    return matchID;

The Extended Target window (XTarget) is EverQuest's aggro list. Any mob that has aggro on you or your group appears in this list. The list contains the spawn ID of each entry, making it a useful source for resolving ambiguous names.

The filter logic is careful about what counts as an NPC:

  • Empty slots and slots with zero spawn ID are skipped.
  • Names must match the target name from the chat line.
  • The spawn must be of type SPAWN_NPC. Players are excluded because you should not be dealing damage to players in normal PvE.
  • NPC pets owned by players are excluded: if MasterID is non-zero and the master is a SPAWN_PLAYER, skip it. This prevents attributing your hits on an enemy necromancer's pet to the wrong entry.
  • NPC pets owned by other NPCs are allowed. A ward or summoned mob from an enemy caster is a legitimate target to track.

Why only use this when exactly one match is found? If two NPCs with the same name are both on your XTarget list, we cannot tell which one a given chat line refers to. Using an ambiguous result would be worse than using the synthetic fallback, because it might silently accumulate damage on the wrong spawn. The one-match requirement means the function only asserts certainty when it has it.

When XTarget is most useful: Damage shield damage. The mob that just hit you and triggered your damage shield is on your aggro list (by definition), is probably not your current target (you may be targeting someone else), and its name is in the DS chat line. If it is the only mob with that name on XTarget, this strategy resolves it correctly.

Strategy 6: Synthetic Fallback

return --m_syntheticIDCounter;  // Returns -1, then -2, then -3...

When every other strategy fails, the engine creates a new negative ID. Each call gets a unique negative number, so multiple unresolved hits on "a skeleton" do not all merge into one entry — they become separate synthetic entries.

Synthetic entries are displayed in the UI without the "#id" suffix that real spawn IDs get. The check is data.spawnID > 0 before appending the ID tag:

std::string label = data.spawnID > 0
    ? fmt::format("{} (#{})", data.name, data.spawnID)
    : data.name;  // Synthetic: just the name, no ID tag

In practice, synthetic IDs should be rare in normal gameplay. They occur mainly when fighting DoT targets without a registered cast (e.g., the DoT was cast before tracking started) or when the XTarget list does not have the mob.

Incoming Damage: pLocalPlayer Fallback

After RecordDamage finishes resolving and accumulating an event, it performs one final check: if targetSpawnID <= 0 and the event is an incoming type (HitBy, HitByNonMelee, MissedMe, DirectHeal, or CritHeal), the engine assigns pLocalPlayer->SpawnID as the target spawn ID:

if (record.targetSpawnID <= 0)
{
    bool isIncoming = (record.type == DamageType::HitBy
        || record.type == DamageType::HitByNonMelee
        || record.type == DamageType::MissedMe
        || record.type == DamageType::DirectHeal
        || record.type == DamageType::CritHeal);
    if (isIncoming && pLocalPlayer)
        record.targetSpawnID = pLocalPlayer->SpawnID;
}

This makes sense because incoming damage and heals happen to you. The "target" from the player's perspective is yourself. Assigning the player's own spawn ID ensures that FCT (floating combat text) entries for these events can be anchored to the correct screen position (your character's location in the world), rather than being discarded due to having no valid spawn ID.

This assignment happens after the damage/heal accumulation logic, so it does not affect combat tracking or target maps. It is used exclusively by the fctManager.AddEntry() call that immediately follows. See the Floating Combat Text page for details on how the FCT system uses spawn IDs for bone-anchored projection.

All pointer checks are guarded. Before dereferencing any of the pointers above, the code confirms it is non-null. A null pointer dereference in a MacroQuest plugin crashes the entire EverQuest process. Every code path in ResolveSpawnID checks pTarget != nullptr, pPet != nullptr, pPetTarget != nullptr, and pLocalPC->pExtendedTargetList != nullptr before accessing members.

Realistic Accuracy Expectations

In typical play (single-target combat, or multiple targets with distinct names), spawn resolution is close to 100% accurate. The challenging cases are:

  • Multiple same-named targets simultaneously: Resolution degrades gracefully — direct hits and DoTs you cast while targeting a specific one are attributed correctly; damage shields and pet damage may be merged under the first-seen entry.
  • DoTs cast before tracking started: No cast record exists. Tick damage falls through to existing entry or XTarget scan.
  • Quickly switching targets during melee: The name check prevents misattribution, but if you switch targets and the old target's hit message arrives after the switch, the hit may fall through to the existing entry or XTarget.
  • DoTs on two identical-name mobs: Only the first matching registry entry is used. The second mob's ticks are attributed to the first entry until the first mob despawns and its registry entry is cleaned up.
