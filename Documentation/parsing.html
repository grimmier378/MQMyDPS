Chat Parsing

Last Updated: 2026-04-20

The Problem: All Data Comes From Text

EverQuest's game client sends every combat and healing event as a formatted chat message. There is no separate "damage event API" that gives you a structured object with damage amount, target, and type. Instead you get a line like:

You crush a skeleton for 150 points of damage.

The parser's job is to read that text and produce a DamageRecord struct with type=Melee, attackVerb="crush", targetName="a skeleton", damage=150. Every piece of information must be extracted from the text because that is the only source available.

Why string_view::find() Instead of Regex?

EQ combat messages have completely fixed formats. "YOU crush X for N points of damage" is always exactly that pattern. There are no optional groups, no alternations in the basic case, no ambiguous structure. In this situation, simple string scanning with std::string_view::find() is faster than regex for two reasons:

  • MSVC's std::regex implementation is known to be significantly slower than manual string operations for short fixed patterns.
  • string_view is a non-owning view of the original chat line buffer. No copies are made when scanning for substrings.

The MQ2 Blech event system (used by MQ2 macros for #event lines) could also work, but it involves registering pattern callbacks and has overhead that is not worth it when the parser is called 60 times per second on every incoming chat line.

Parse Flow

Parse(line, color) wraps the raw C string in a string_view (zero-copy), then tries each of the 14 TryParse* methods in order. The first method that returns true wins, and the populated DamageRecord is returned as an std::optional. If no method matches, nullopt is returned and the caller (ProcessChat) ignores the line.

The methods are ordered by expected frequency in normal play: melee hits are by far the most common message, followed by non-melee, then crits, DoTs, misses, and so on. This means on a typical melee hit, the parser returns after the very first check.

Pet methods are special: they are only attempted if the pet name is non-empty. If you have no pet, the parser skips both pet methods entirely, saving two string comparisons on every non-matching line.

How Each Method Works Internally

TryParseMelee

Checks that the line starts with "You ". Finds " for " and " points of damage" to locate the damage number. Everything between "You " and " for " is the verb-and-target substring, which is split at the first space to separate verb from target name.

"You crush a skeleton for 150 points of damage."
 |   |verb||--target--|     |damage|
 "You " prefix confirmed
 verb = "crush"
 targetName = "a skeleton"
 damage = 150

TryParseNonMelee

Requires the character name to be set (via SetCharName). Checks that the line starts with "[CharName] hit ". The target name sits between the prefix and " for ". The damage is after " for " and before " points of non-melee damage".

Why does this need the character name? Because EQ writes it as "Grimmier hit X for N" rather than "You hit X for N", using your actual name. Without knowing the name, you cannot identify which lines are yours versus another player standing nearby.

TryParseCrit and TryParseDeadlyStrike

Crit messages in EQ do not include the target name in the standard format. They just say "You score a critical hit! (1200)". The damage is extracted from inside the parentheses using rfind('(') and rfind(')'). The target name is left empty by the parser; it is filled in later by RecordDamage by reading pTarget->DisplayedName.

TryParseDoT

Extracts three pieces of information: target name (before " has taken "), damage (between " has taken " and " damage from your "), and spell name (after " damage from your ", with trailing period stripped). The spell name is critical for the DoT registry — it is how the engine matches a tick back to the specific cast that caused it.

TryParseDamageShield

Damage shield messages say "[mob] was hit by non-melee for N points of damage." The target name is everything before " was hit by non-melee for ". Note that the parser does not need your character name here because the format does not include it.

TryParseCritHeal

Matches the "You perform an exceptional heal!" line and extracts the bonus amount from inside parentheses using rfind. This produces a DamageType::CritHeal record. Note that crit heals do not carry a target name — EQ does not include one in the message. The engine tracks battleCritHeals as a battle-level total, not per heal target.

TryParseDirectHeal

This is the newest parser method and the only one that handles two distinct EQ message formats. Both are produced when your healing spells land:

Format A — basic:

You have healed Grimmier for 2500 points.

Format B — with spell name:

You have healed Grimmier for 2500 hit points with your Celestial Remedy.

The method works as follows:

  1. Confirms the line starts with "You have healed ".
  2. Finds " for " to locate the start of the number and end of the player name.
  3. Extracts the target name from between the prefix and " for ".
  4. Checks for " hit points" (Format B marker) first. If found, extracts the number between " for " and " hit points", then looks for " with your " after the number to capture the spell name (stripping the trailing period).
  5. If " hit points" is not found, checks for " points." (Format A) and extracts the number between " for " and " points.".
  6. If neither marker is found, returns false (not a recognized heal line).

The resulting record has type=DirectHeal, attackVerb="heal", targetName set to the player name, damage set to the heal amount, and optionally spellName set if Format B was matched. The spell name is stored in DamageRecord::spellName for future use but is not currently used for attribution (heals are keyed by player name, not spell name).

TryParsePetMelee and TryParsePetNonMelee

Both require the pet name to be set and non-empty. The pet name is set in ProcessChat on every call by checking pLocalPlayer->PetID and looking up the pet's DisplayedName. This ensures that if you resummon a pet with a different name, the parser adapts immediately on the next chat line.

The attack verb for pet melee is stored as "[PetName] [verb]" (e.g. "Gobber hits"). This allows the combat spam overlay to show pet hits distinctly from your own hits, even though they share the "hit" color key.

Dynamic Names

The parser uses two dynamic names set at runtime:

  • SetCharName(name) — your character name, set once when settings load in SetGameState(GAMESTATE_INGAME). Used by TryParseNonMelee and TryParseDeadlyStrike, which include your actual name rather than "You".
  • SetPetName(name) — your pet's current display name. Updated on every call to ProcessChat, not just on load. Pets can be killed and resummoned within a single play session, potentially with different names, and this ensures the parser always has the current name.

Number Extraction

ExtractNumber(text, start, end) scans a substring for a number using std::from_chars, which is the fastest standard-library integer parser available. It reads directly from the string's memory without any allocations. The function returns int64_t so it can handle arbitrarily large damage values (important for high-end content or parser testing).

The function also has boundary validation: if start >= end or end > text.size(), it returns 0 safely rather than causing a buffer overread. This matters because the calling methods compute these positions from find() results, and a malformed or unexpected line could theoretically produce out-of-bounds positions.

Filtering in ProcessChat

After parsing succeeds, ProcessChat() applies the user's visibility settings before calling RecordDamage(). This filtering happens in the engine, not the parser, because the parser's job is only to recognize patterns — decisions about what to track are the engine's responsibility.

  • Miss records dropped if showMyMisses is false
  • MissedMe dropped if showMissMe is false
  • HitBy and HitByNonMelee dropped if showHitMe is false
  • CritHeal dropped if showCritHeals is false
  • DamageShield dropped if showDS is false
  • Pet damage is only parsed at all if addPet is true; if false, the pet name is cleared to an empty string, which causes TryParsePetMelee and TryParsePetNonMelee to return false immediately
  • DirectHeal events are not filtered by any setting — direct heals are always recorded when they arrive. They are excluded from damage counters automatically by the isDamageOut check in RecordDamage.
Why filter in the engine rather than the parser? The parser is meant to be a pure text-to-structure converter. It has no knowledge of user preferences. Keeping settings out of the parser means you could reuse the same parser in a different context (a test harness, a log file reader) without any behavior changes. The engine is the appropriate place for policy decisions.
