Plugin Lifecycle

Last Updated: 2026-04-20

What Is a MacroQuest Plugin Lifecycle?

MacroQuest loads your plugin as a DLL into the EverQuest process. It calls a set of well-known exported functions (callbacks) at specific moments: when the plugin loads, when the game state changes, when a chat line arrives, 60 times per second on pulse, and when a mob despawns. Your plugin registers these callbacks and MacroQuest calls them at the right times. Understanding which callback fires when is essential to knowing why the code is structured the way it is.

Plugin Callbacks

CallbackWhen it firesWhat MQMyDPS does
InitializePlugin Once, when the plugin DLL is loaded (either at EQ startup if auto-loaded, or when the user runs /plugin MQMyDPS). Creates the global MyDPSEngine on the heap, calls Initialize() (creates the parser, seeds default colors, sets session start time), registers the /mydps command, registers the settings panel at "plugins/MyDPS" in the MacroQuest Settings window, and registers the ${MyDPS} TLO.
ShutdownPlugin Once, when the plugin is unloaded or EQ exits. Unregisters TLO, settings panel, and command (in that order — TLO first because macros might reference it), calls Shutdown() (saves settings, destroys parser), deletes engine, nulls the pointer.
SetGameState Whenever the game state changes: entering world, going to character select, zoning, etc. GAMESTATE_INGAME: calls LoadCharacterSettings(), which reads per-character INI settings, sets character and pet names on the parser, checks IsPluginLoaded("MQMyChat") and caches the result in m_myChatLoaded, and starts auto-tracking if AutoStart=true. GAMESTATE_CHARSELECT: saves settings and clears character/server name fields.
OnIncomingChat Every chat line received by the EQ client, including combat messages. Returns bool — returning true would suppress the line from chat display. MQMyDPS always returns false (never suppresses). If tracking is enabled, feeds the chat line to ProcessChat() for parsing and damage or heal recording.
OnPulse Every game frame (approximately 60 times per second). MQMyDPS throttles its own work to 250ms intervals using a static timer. Calls TrackDoTCasting() to detect spell cast completions, UpdateCombatState() to check if combat ended, CleanExpiredRecords() to prune old spam entries, and RefreshAggregates() to sort cached session target and heal aggregates when the aggregateDirty flag is set. Does nothing if tracking is disabled.
OnUpdateImGui Every render frame, inside ImGui's render pass. Only when GAMESTATE_INGAME. Calls RenderMainWindow, RenderConfigWindow, RenderCombatSpam, and RenderFloatingText on the renderer. Each function checks its own visibility flag first, so invisible windows cost almost nothing. RenderCombatSpam, RenderMainWindow, and RenderConfigWindow apply and reset the active theme via ImGuiTheme before/after ImGui::Begin/ImGui::End.
OnBeginZone When the player begins zoning (before the zone load). First calls SaveCharacterSettings() to persist current settings to INI, then clears inCombat flag, the entire activeDoTs registry, and calls fctManager.Clear() to remove all floating combat text entries. All spawn IDs from the old zone are now invalid, so DoT attribution and FCT must start fresh in the new zone.
OnRemoveSpawn When any spawn (mob, player, pet) is removed from the world, including when a mob dies, despawns, or you lose sight of it. Removes all ActiveDoT entries whose targetSpawnID matches the despawned spawn. This prevents stale DoT entries from mis-attributing future hits on a new mob that happens to reuse the same name.

Why 250ms Throttling in OnPulse?

OnPulse fires every frame, which at 60 FPS means 60 calls per second. The work done in TrackDoTCasting and UpdateCombatState is not expensive, but it reads live game state pointers (pLocalPlayer->CastingData, GetCombatState()) and does timer comparisons. Doing this 60 times per second provides no benefit over doing it 4 times per second for DPS tracking purposes. The 250ms throttle is implemented with a static local timer variable that persists between calls:

static auto pulseTimer = std::chrono::steady_clock::now();
if (std::chrono::steady_clock::now() < pulseTimer)
    return;  // Too soon, skip this frame
pulseTimer = std::chrono::steady_clock::now() + std::chrono::milliseconds(250);

Combat State Machine

The state machine governs when battle counters are active and when a battle is finalized into history. Understanding this helps explain why damage might appear in one battle vs. another, and why there is a delay before a battle closes out.

[Not Tracking] --/mydps start--> [Tracking, Out of Combat] | damage event with isDamageOut=true (your melee, spell, DoT, pet, or DS that actually landed - not a miss) | v [In Combat] (battleDamage accumulating, currentTargets populating, currentHealTargets populating when heals land, battlePetDamage/battleNonMeleeDmg tracking, battleDirectHeals/battleCritHeals tracking) | GetCombatState() != eCombatState_Combat (game says you left combat - but we wait) | v [Combat End Pending] (m_combatEndPending=true, m_leftCombatTime set) | wait battleEndDelay seconds (default 10s, configurable) If you hit something again, return to [In Combat] | v FinalizeBattle() (BattleData created, currentTargets moved to battle.targets, currentHealTargets moved to battle.healTargets, summary sent to MQMyChat if loaded, all battle counters reset to 0) | v [Tracking, Out of Combat]

Why Combat Starts on First Hit, Not on GetCombatState()

The engine detects combat entry by watching for isDamageOut=true in RecordDamage, not by polling GetCombatState(). This is because the combat spam overlay and battle counters should start at the moment you deal your first damage, which is a well-defined point in time associated with the actual event. GetCombatState() is only used to detect combat end, because EQ's combat flag transitions after your last hit, giving a reliable "you are no longer fighting anything" signal.

Note that heal events (DirectHeal, CritHeal) and incoming damage events (HitBy, HitByNonMelee, MissedMe) do not trigger combat entry on their own. isDamageOut excludes all five of these types explicitly. Heals only contribute to the battle totals if you are already in combat (i.e., you dealt damage first).

Why the Battle End Delay Exists

GetCombatState() can briefly return non-combat during pauses in a fight: between mob spawns in a wave, while running to the next pull, or when a DoT is ticking but you have not swung recently. Without the delay, each such pause would finalize the current battle and start a new one, fragmenting what is logically a single combat session into many tiny battles. The 10-second default means a pause shorter than 10 seconds is treated as part of the same battle.

Battle Finalization

FinalizeBattle() creates a BattleData snapshot:

  1. Increments battleCounter and stores it as battleNumber
  2. Copies battle counters into the battle struct, including battlePetDamage, battleNonMeleeDmg, battleDirectHeals, and battleCritHeals
  3. Computes DPS and average damage
  4. Moves currentTargets into battle.targets via std::move (zero-cost transfer of the hash map's heap memory)
  5. Moves currentHealTargets into battle.healTargets via std::move
  6. Sends a colored summary to MQMyChat if m_myChatLoaded is true, listing per-target DPS breakdown and per-player heal totals
  7. Pushes the completed battle to battleHistory
  8. Sets inCombat = false and m_combatEndPending = false
  9. Resets all battle counters (damage, heals, hits) to zero
  10. Clears currentTargets and currentHealTargets (both were emptied by the move, but explicitly cleared for safety)
Session damage vs battle damage: sessionDamage and sessionHitCount are never reset by FinalizeBattle(). They accumulate across all battles for the entire session (until ResetAll() is called or the plugin is unloaded). This is why ${MyDPS.DPS} gives your lifetime session DPS while ${MyDPS.BattleDPS} gives the current battle DPS.

Settings Persistence

Settings are stored as INI files at [MQ Config Dir]\MQMyDPS\[Server]\[CharName].ini. The directory is created automatically on first save via std::filesystem::create_directories(). This per-server, per-character path means you can have different display preferences on different characters or servers without them interfering with each other.

Load/Save Timing

  • LoadCharacterSettings() is called when SetGameState fires with GAMESTATE_INGAME. At this point pLocalPC is valid and the character and server names can be read. This is also where m_myChatLoaded is set by calling IsPluginLoaded("MQMyChat").
  • SaveCharacterSettings() is called in Shutdown() (plugin unload), in UnloadCharacterSettings() (char select), and in OnBeginZone() (before clearing combat state).
  • The Save button in the config window also calls SaveCharacterSettings() immediately and closes the config window.
  • Save-on-close: When any window (main, config, or combat spam) is closed via the X button, the renderer detects the visibility transition and calls SaveCharacterSettings() to persist the new window state.

recentRecords Cleanup

CleanExpiredRecords() runs every 250ms in OnPulse. It pops records from the front of the recentRecords deque whose age exceeds settings.displayTime seconds. Because records are appended in timestamp order, the front is always the oldest, so this is an efficient O(n) scan that stops at the first record that is still fresh.

while (!recentRecords.empty())
{
    auto age = now - recentRecords.front().timestamp;
    if (age > maxAge)
        recentRecords.pop_front();
    else
        break;
}
Previous behavior: An earlier version of this cleanup used displayTime * 3 as the maximum age, keeping records around three times longer than they were visible. The current implementation uses displayTime directly — records are cleaned as soon as they would no longer be displayed by the spam overlay.
